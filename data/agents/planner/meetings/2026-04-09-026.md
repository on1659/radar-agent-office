#### L1→L2 예상 블로커 목록

**블로커: 1개 (Medium)**

| 항목 | 상세 | 심각도 |
|------|------|--------|
| 실제 에이전트 ID에 스프라이트 없음 | `Character.ts:69` — `PIXEL_DATA[agentId] ?? null`. 하드코딩된 8개 데모 ID(`jimin`/`hyunwoo` 등) 외의 실제 에이전트 ID는 `pixelChar = null` → `draw()` 조기 리턴 → Canvas에 아무것도 안 그려짐 | Medium |

**페이지 크래시는 아닙니다.** OfficePage 자체는 로드되고, Canvas도 초기화되고, 타일맵도 그려집니다. 다만 **실제 워크스페이스의 에이전트가 Canvas에 표시되지 않습니다.** 빈 오피스에 타일만 보이는 상태입니다.

이건 L2 판정에서 미묘합니다. "런타임 에러 0"이라는 기준으로는 **L2 통과**입니다 — 에러 없이 렌더링됩니다. 하지만 "에이전트가 보여야 한다"는 기준으로는 **L2 미달**입니다. 하은 씨가 OfficePage 성공 기준을 정의할 때 이 구분을 명확히 해야 합니다.

**비블로커:**
- Empty workspace 시 demo 에이전트 표시 (`OfficePage:36-39` — wsAgents와 skills 모두 없으면 8개 하드코딩 ID 폴백). 워크스페이스 연결 전에는 데모가 보임
- `engine?.getCamera().isDragging` — 옵셔널 체이닝 이미 적용. null-safe

#### useGameEngine 브릿지 품질 3줄 요약

**설계 완성도: 높음. Phase 2 수정 필요 없음.**

1. **React→Canvas 브릿지가 이미 존재한다.** `useGameEngine` 훅이 `agentStatuses: Record<string, AgentStatus>`를 입력으로 받아 `useEffect`에서 `engine.updateAllAgentStatuses()`를 호출하는 패턴이 구현 완료. 이전 회의에서 "브릿지 설계가 Phase 2의 핵심 기술 과제"라고 했지만, **이미 해결된 문제다.**

2. **프레임 동기화 설계가 깔끔하다.** `onPositionUpdate` 콜백 → `setFrameTick` → `speechBubbles` useMemo 재계산 패턴으로, RAF 루프와 React 렌더링 사이클이 분리된 채로 말풍선 위치를 정확히 동기화한다.

3. **게임엔진 격리 원칙 준수.** `game/` 전체에서 React import 없음. Camera, TileMap, Character, Pathfinder 모두 순수 TypeScript + Canvas API.

#### game/ 재사용 범위 판정

**전부 재사용 가능. 새로 만들 것 없음.**

| 모듈 | 상태 | Phase 2 수정 필요 |
|------|------|-------------------|
| `GameEngine.ts` | ✅ 완전 작동. `updateAllAgentStatuses` API 존재 | ❌ 없음 |
| `TileMap.ts` | ✅ 20×14 오피스 레이아웃, 캐시 렌더링, `isWalkable` | ❌ 없음 |
| `Character.ts` | ✅ A* 이동, 방향별 스프라이트, 상태별 시각 효과 | ⚠️ 실제 에이전트 ID용 픽셀 데이터 추가 필요 |
| `Camera.ts` | ✅ 마우스 드래그 패닝, 휠 스크롤, 경계 클램핑 | ❌ 없음 |
| `Pathfinder.ts` | ✅ A* 알고리즘, TileMap 연동 | ❌ 없음 |

**유일한 수정 대상은 `Character.ts`의 `PIXEL_DATA`입니다.** 실제 에이전트 ID에 대한 기본 스프라이트(또는 fallback 스프라이트)를 추가하면 됩니다. 이건 `PIXEL_DATA[agentId] ?? DEFAULT_PIXEL_DATA` 한 줄로 해결 가능합니다.

**2. `npm run dev` → OfficePage 런타임 에러 — Meeting #25 execution에서 직접 기동하지는 못했습니다.**

코드 레벨 확인은 완료했지만, `npm run dev` 실행은 하지 못했습니다. 다만 코드 분석 기반으로 런타임 에러 예측을 드립니다:

**3. 서진 씨 보고에 대한 분석 — 핵심 발견 3가지를 Plan에 어떻게 반영하는가.**

**발견 1: Character.ts PIXEL_DATA fallback 미존재 (Medium 블로커)**

이 경우엔 어떻게 되나요? 시나리오를 정리합니다:

| 시나리오 | 현재 동작 | 사용자 경험 |
|----------|----------|-----------|
| 워크스페이스에 에이전트 3개 | Canvas에 아무것도 안 보임 (PIXEL_DATA에 해당 ID 없음) | 빈 오피스. **"고장났나?"** |
| 워크스페이스 미연결 (빈 상태) | 데모 에이전트 8개 표시 | 정상적으로 보임 |
| 워크스페이스에 에이전트 0개 | 데모 에이전트 8개 표시 | 정상적으로 보임 |

**역설적 상황입니다.** 에이전트가 없으면 데모가 보이고, 에이전트가 있으면 안 보입니다. 사용자 입장에서 "에이전트를 연결했더니 오히려 빈 화면이 됐다"는 **후퇴 경험**입니다.

서진 씨가 "결함 수정 범주"라고 한 것에 동의합니다. **이건 Step 1 결함 수정에 반드시 포함되어야 합니다.** Plan Section 2.1에 명시하겠습니다.

**발견 2: useGameEngine 브릿지 완성, game/ 전부 재사용**

Plan 관점에서 이건 **Phase 2.0의 기술적 리스크를 크게 낮춥니다.** Section 5 Risk에서:

| 리스크 | Meeting #25 상태 | 서진 확인 후 |
|--------|-----------------|-------------|
| React→Canvas 브릿지 설계 | ~~Medium~~ → **Closed** | 이미 구현 완료 |
| 게임엔진 VS Code 잔재 | ~~Low~~ → **Closed** | 잔재 0건 |
| Canvas 접근성 한계 | **High** 유지 | 변화 없음 |
| **신규: PIXEL_DATA fallback 미존재** | — | **Medium** 추가 |

**리스크 2개 해소, 1개 신규 추가.** 순리스크 감소입니다. Phase 2가 예상보다 가벼워졌습니다.

**발견 3: L2 판정 기준 모호성**

서진 씨가 정확히 짚었습니다:

> "런타임 에러 0이라는 기준으로는 L2 통과. 에이전트가 보여야 한다는 기준으로는 L2 미달."

**이건 성공 기준 정의의 핵심 문제입니다.** L2를 어떻게 정의하느냐에 따라 Phase 2 Step 1의 범위가 달라집니다.

제 제안 — **L2를 두 단계로 분리합니다:**

| 수준 | 정의 | OfficePage 판정 기준 |
|------|------|---------------------|
| L2a | 런타임 에러 0 + 페이지 로드 | 콘솔 에러 없음 + Canvas 초기화 성공 |
| L2b | 핵심 기능 동작 | 실제 에이전트가 Canvas에 표시됨 + 상태 변경 반영 |

L2a는 Character.ts fallback 없이도 통과합니다. L2b는 fallback이 있어야 통과합니다. **Step 1의 게이트는 L2b여야 합니다.** "에러 없이 빈 화면"은 사용자에게 "에러 있는 화면"과 다를 바 없습니다.

다만 — 이 L2a/L2b 구분이 프레임워크를 과도하게 복잡하게 만든다면, 민준 씨 판단으로 단순화해도 됩니다. **중요한 건 "에이전트가 보여야 L2"라는 기준을 명시하는 겁니다.**

**5. 서진 씨 "킥오프 전 Character.ts 선행 작업" 제안에 대해.**

동의합니다. **단, 조건이 있습니다.**

**이 한 가지만 결정되면 서진 씨가 선행 작업 가능합니다.**

| A. 기존 캐릭터 복제 | 데모 중 하나(예: `jimin`)를 기본으로 | 구현 0초 | 모든 실제 에이전트가 같은 모습 — "누가 누군지 구분 안 됨" |
| B. 단색 실루엣 | 회색 사람 형태 아웃라인 | 구분 가능 — "이건 기본 캐릭터다" | 고유 캐릭터와 시각적 위계 다름 — 2등 시민 느낌 |
| **C. 해시 기반 색상 + 기본 형태** | 에이전트 ID를 해시 → 색상 결정, 동일한 기본 실루엣 | **자동 구분 + 일관된 형태** | 구현 5분 더 걸림 |

**옵션 C를 권장합니다.**

이유:
- 에이전트가 3~10개면 "전부 같은 모습"(A)은 즉시 불편해집니다. Canvas에서 에이전트를 구분하는 유일한 단서가 색상이어야 합니다.
- 해시 기반 색상은 에이전트 ID가 같으면 항상 같은 색이 나오므로 **세션 간 일관성**이 있습니다. 랜덤이 아닙니다.
- 기본 실루엣은 데모 캐릭터보다 단순한 형태(예: 3×5 사각형 사람 아웃라인)로 — **"이건 커스텀 캐릭터가 아직 없는 에이전트"**라는 시각적 신호를 줍니다.

**구현 스펙:**
```typescript
// Character.ts — DEFAULT_PIXEL_DATA 생성
function getDefaultPixelData(agentId: string): PixelData {
  const hash = agentId.split('').reduce((h, c) => (h * 31 + c.charCodeAt(0)) | 0, 0);
  const hue = Math.abs(hash) % 360;
  const baseColor = `hsl(${hue}, 60%, 55%)`;
  // 기본 실루엣: 머리(1px) + 몸통(3px) 형태
  return { ...BASE_SILHOUETTE, primaryColor: baseColor };
}
```

서진 씨 — `PIXEL_DATA[agentId] ?? getDefaultPixelData(agentId)` 형태가 됩니다. "한 줄"이라기보다 함수 하나 추가입니다. 하지만 5분이면 됩니다.
