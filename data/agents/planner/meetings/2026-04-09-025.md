**2. 서진 씨 L1/L2/L3 프레임워크 — 좋은 구조입니다. 한 가지 보완합니다.**

| 수준 | 설명 | 검증 방법 |
|------|------|-----------|
| L1 | 존재 + 빌드 통과 | `tsc --noEmit` + `npm run build` |

**4. 서진 씨 Phase 2 순서에 대해 — 한 가지 조정.**

서진 씨: "검증 먼저, 보강 다음"에 동의합니다. 하지만 **접근성을 "보강 단계"로 분류하면 안 됩니다.**

접근성 2개 항목:
- `prefers-reduced-motion` — **기존 코드 수정**. StatusBadge.tsx의 `radar-pulse` keyframe에 미적용.
- StatusBadge `role="status"` + `aria-label` — **기존 컴포넌트 보강**.

이건 "새 기능 추가"가 아니라 **"기존 코드의 결함 수정"**입니다. reduced-motion 미대응은 접근성 표준 위반이고, ARIA 속성 누락은 스크린 리더 사용자에게 정보를 전달하지 못하는 버그입니다.

**제안: 접근성 2개를 "보강 단계"에서 "검증 단계"로 올립니다.** L2 검증과 병렬로 처리 가능합니다. 코드 변경량이 적고(StatusBadge.tsx 수정 + variables.css에 이미 있는 reduced-motion override를 radar-pulse에 확장), 다른 작업에 의존하지 않습니다.

```
[Phase 2.0 — Step 1: 현상 파악 + 결함 수정]
  서진: OfficePage L2 검증 (npm run dev)
  서진: 접근성 2개 수정 (reduced-motion + ARIA) ← 병렬
  하은: OfficePage 성공 기준 정의

[Phase 2.0 — Step 2: 갭 측정]
  gap-detector → OfficePage 설계 vs 코드
  Step 2 렌더링 검증

[Phase 2.0 — Step 3: 보강]
  갭 해소
  Canvas 접근성 전략 (이건 진짜 새 작업)
```

Canvas 접근성(옵션 2: 보조 뷰 포지셔닝)은 새 설계가 필요하므로 보강 단계가 맞습니다. 하지만 StatusBadge 접근성은 결함 수정입니다.

| 접근성 수준 | 스크린 리더 완전 지원 | OfficePage는 비접근성, AgentsPage로 우회 |

**옵션 2를 최종 권장합니다.**

이유: OfficePage는 "시각적 보조 뷰"입니다. 같은 데이터를 AgentsPage(접근 가능한 리스트 뷰)에서 이미 제공하고 있습니다. Canvas에 DOM 레이어를 덮으면 게임엔진 격리 원칙(`game/은 순수 Canvas`)이 깨지고, 60fps 유지가 어려워집니다. **접근성을 위해 성능을 희생하는 건 양쪽 사용자 모두에게 나쁜 결과입니다.**

대신 OfficePage에 `aria-label="에이전트 상태를 시각적으로 표현하는 오피스 뷰입니다. 상세 정보는 에이전트 목록 페이지를 이용하세요."` 한 줄을 Canvas 컨테이너에 넣으면 됩니다. 스크린 리더 사용자에게 "여기는 시각 전용이고, 리스트 뷰가 접근 가능한 대안"임을 안내합니다.
