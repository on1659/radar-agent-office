**Phase 2 Plan 초안에 대해 — 하은 씨가 이미 작성했다는 게 좋은 소식입니다.**

서진 씨 기술 과제 지적에 전적으로 동의합니다. "OfficePage 기본 렌더링"이라는 한 줄 뒤에 숨어 있는 복잡도 — **React와 Canvas의 경계 설계**가 Phase 2의 가장 큰 UX 과제입니다.

UI 관점에서 구체적으로 말하면:

**AgentsPage ↔ OfficePage 전환 시 "어떤 상태가 유지되어야 하는가"**가 핵심입니다.

- 사용자가 AgentsPage에서 에이전트 하나를 선택하고 → OfficePage로 전환하면 → 그 에이전트가 캔버스에서 하이라이트되어야 하는가?
- OfficePage에서 에이전트 스프라이트를 클릭하면 → AgentsPage로 전환 시 → 해당 에이전트가 선택된 상태여야 하는가?
- 전환 자체는 라우팅이라 단순하지만, **선택 상태 동기화**가 두 뷰를 묶는 고리입니다.

이건 Plan 리뷰에서 다룰 사항입니다. 지금은 Step 1부터.

**접근성 스펙 업데이트.**

Meeting #22에서 작성한 접근성 설계 문서(`docs/02-design/features/a11y-agent-status.design.md`)가 있습니다. 추가로 정리한 내용:

**8px 아이콘 가독성 — 결론이 나왔습니다.**

Meeting #22 execution에서 코드를 확인했을 때 발견한 사실: **StatusBadge.tsx에 이미 텍스트 레이블(`{s.label}`: "Idle"/"Working"/"Error"/"Queued")이 존재합니다.** 색각 이상 사용자도 현재 텍스트로 상태를 구분할 수 있습니다.

8px 유니코드 아이콘(○, ▶, ✕, ⏳)은 **가독성이 부적합합니다.** 8px에서 ○과 ●의 차이를 구분하기 어렵고, ⏳은 16px 이상이 필요합니다. 기존 텍스트 레이블이 아이콘보다 더 나은 접근성을 제공합니다.

**권장: 아이콘 추가 없이 기존 텍스트 레이블 유지.** 접근성 Phase 2 비기능 요구사항에서 다룰 항목은:
1. `prefers-reduced-motion` 대응 — `radar-pulse` keyframe에 미적용 상태
2. Canvas 접근성 전략 — 숨겨진 DOM 레이어 vs 보조 뷰 포지셔닝

아이콘 추가는 **중복이고 불필요합니다.** 이 결론을 Phase 2 Plan 리뷰에 반영해주세요.
