**1. Phase 1.5 백로그 스펙 — 항목별로 다릅니다.**

**Step 1 — 에이전트 없이 확인 (이번 회의)**

|------|------|-----------|

| 2 | 레이아웃 | 좌 1/3 : 우 2/3 비율 시각 확인 |

| 펄싱 애니메이션 + reduced-motion | working 에이전트 존재 |

**2. 접근성 스펙 — 솔직히 말하면 미완성입니다.**

하은 씨가 정확하게 짚었습니다. "아이콘이나 텍스트 레이블이 답"은 방향이지 스펙이 아닙니다. 구체적 방안을 정리하겠습니다.

**접근성 초안 — StatusBadge 아이콘 방안:**

| 상태 | 현재 (색상만) | 제안 (아이콘 추가) | 근거 |
|------|-------------|-------------------|------|
| idle | 회색 ● | `—` (대시) 또는 빈 원 ○ | 비활성 상태를 형태로 표현 |
| working | 파란 ● + pulse | `▶` (재생) 또는 회전 아이콘 | 동작 중을 형태로 표현 |
| error | 빨간 ● | `✕` (엑스) | 실패를 형태로 표현 |
| queued | 노란 ● | `⏳` 또는 `…` (말줄임) | 대기를 형태로 표현 |

**적용 위치**: 기존 StatusBadge 컴포넌트 내부. 색상 점 옆에 8×8 크기 아이콘 또는 유니코드 문자. 별도 컴포넌트 분리 불필요 — StatusBadge가 이미 상태별 렌더링을 담당하고 있으니까 거기서 확장하는 게 자연스럽습니다.

**성공 기준**: 모노크롬(흑백) 상태에서 4개 상태를 아이콘 형태만으로 구분할 수 있는가.

**다만 이건 완성된 스펙이 아닙니다.** 실제로 렌더링해보고 8px에서 아이콘이 읽히는지, 점과 아이콘이 함께 있을 때 시각적으로 깔끔한지 확인해야 합니다. 이건 Phase 2 설계 단계에서 프로토타입으로 검증할 항목입니다.

**Phase 2 방향 — 하은 씨 제안에 동의합니다.**

"Phase 1.5를 건너뛴다"가 아니라 **"Phase 2에 흡수한다"**가 정확한 표현입니다. 서진 씨와 하은 씨가 도달하려는 곳은 같은데 — 하은 씨 프레이밍이 더 정확합니다. 접근성이 Phase 2 설계 문서의 비기능 요구사항으로 들어가면, "건너뛴" 게 아니라 "정식으로 다뤄지는" 겁니다.

**하은 씨 Phase 2.0 범위 초안 3개에 UI 관점 의견:**

| 항목 | 판단 |
|------|------|
| OfficePage 기본 렌더링 (Canvas 타일맵) | **핵심.** 게임엔진 격리 원칙이 있으므로 React와 Canvas 경계가 명확해야 함. 이 경계 설계가 Phase 2의 가장 큰 UX 과제 |
| AgentsPage ↔ OfficePage 이중 뷰 | **필수.** 같은 데이터, 다른 표현. 뷰 전환 시 상태 유지(선택된 에이전트 등)가 핵심 UX |
| 접근성 비기능 요구사항 | **동의.** StatusBadge + OfficePage 양쪽 동시 설계. OfficePage 캔버스에서는 색상 외에 스프라이트 형태 자체로 상태를 구분해야 함 — 이건 접근성과 게임 디자인이 겹치는 지점 |

한 가지 추가 — **OfficePage 캔버스에서의 접근성은 StatusBadge보다 복잡합니다.** Canvas는 DOM이 아니라 픽셀 렌더링이에요. 스크린 리더가 읽을 수 없습니다. ARIA 속성도 붙일 수 없습니다. Canvas 옆에 **접근 가능한 텍스트 레이어**(숨겨진 DOM 요소)를 두거나, OfficePage 자체를 "시각적 보조 뷰"로 포지셔닝하고 AgentsPage를 "접근 가능한 기본 뷰"로 두는 방법이 있습니다.

이건 Phase 2 설계에서 논의할 사항이지 지금 결정할 건 아닙니다. **기록만 해두겠습니다.**
